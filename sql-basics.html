SQL Basics

From SELECT to window functions. Each concept shown with a real example — not a fake employees table you'll never query in real life.

The basics that actually matter

SQL is declarative: you describe what you want, not how to get it. The database figures out the execution plan. This is either liberating or terrifying depending on how much you trust your query optimiser.

-- The shape of a query:
SELECT column1, column2
FROM table_name
WHERE condition
GROUP BY column1
HAVING aggregate_condition
ORDER BY column1 DESC
LIMIT 100;

These clauses execute in a specific order: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT. Understanding this order explains why you can't use a SELECT alias in a WHERE clause — the alias doesn't exist yet when WHERE is evaluated.

JOINs

Joins combine rows from two tables based on a related column.

-- Find all orders with their customer names
SELECT orders.id, orders.total, users.name
FROM orders
INNER JOIN users ON orders.user_id = users.id
WHERE orders.created_at > NOW() - INTERVAL '7 days';
  • INNER JOIN — only rows with a match in both tables
  • LEFT JOIN — all rows from the left table, NULLs where no match on the right
  • RIGHT JOIN — all rows from the right table (rare; usually rewrite as a LEFT JOIN)
  • FULL OUTER JOIN — all rows from both tables

When you LEFT JOIN and filter on a column from the right table in the WHERE clause, you accidentally turn it into an INNER JOIN — because rows with NULL in that column get filtered out. Filter on right-table conditions in the ON clause instead, or use WHERE right_table.col IS NULL intentionally.

Aggregations and GROUP BY

-- Daily signups for the last 30 days
SELECT
  DATE_TRUNC('day', created_at) AS day,
  COUNT(*) AS signups,
  COUNT(*) FILTER (WHERE plan = 'pro') AS pro_signups
FROM users
WHERE created_at > NOW() - INTERVAL '30 days'
GROUP BY 1
ORDER BY 1;

Every column in SELECT must either be in GROUP BY or wrapped in an aggregate function (COUNT, SUM, AVG, MIN, MAX). The database needs to know how to collapse multiple rows into one.

CTEs — Common Table Expressions

CTEs let you name a subquery and reference it like a table. They make complex queries readable.

WITH active_users AS (
  SELECT user_id
  FROM sessions
  WHERE started_at > NOW() - INTERVAL '30 days'
  GROUP BY user_id
),
revenue AS (
  SELECT user_id, SUM(amount) AS total
  FROM payments
  GROUP BY user_id
)
SELECT
  active_users.user_id,
  COALESCE(revenue.total, 0) AS revenue_30d
FROM active_users
LEFT JOIN revenue USING (user_id);

Window functions

Window functions compute a value across a set of rows related to the current row, without collapsing them into a single row like GROUP BY does.

-- Rank users by revenue within each country
SELECT
  user_id,
  country,
  revenue,
  RANK() OVER (PARTITION BY country ORDER BY revenue DESC) AS country_rank,
  SUM(revenue) OVER (PARTITION BY country) AS country_total
FROM user_revenue;

The most useful window functions: ROW_NUMBER(), RANK(), LAG()/LEAD() (previous/next row value), SUM() OVER, AVG() OVER.

Performance gotchas SELECT * is fine for exploration, slow in production — you're fetching columns you don't need. LIKE '%term%' with a leading wildcard can't use an index. NOT IN (subquery) has surprising behaviour when the subquery returns NULLs — use NOT EXISTS instead. Always check EXPLAIN ANALYZE on slow queries before adding indexes randomly.

Indexes

An index lets the database find rows without scanning the whole table. Creating one is a one-liner; knowing when to create one takes judgement.

CREATE INDEX idx_orders_user_created
ON orders (user_id, created_at DESC);

Indexes speed up reads and slow down writes (the index must be updated on every INSERT/UPDATE/DELETE). Index columns you filter on frequently, columns used in JOINs, and columns you ORDER BY. Don't index every column — the write overhead adds up, and the query planner can only use so many indexes per query.
