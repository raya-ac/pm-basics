API Design

REST, GraphQL, and the principles behind APIs that don't make engineers cry.

What Is an API?

An API (Application Programming Interface) is a defined contract that lets one piece of software talk to another. It specifies what requests can be made, in what format, and what responses to expect — without exposing the underlying implementation.

In web development, APIs almost always mean HTTP APIs: one service sends an HTTP request to another (or to a server) and gets a structured response back, typically as JSON. Your mobile app calls an API to fetch user data. Your checkout page calls an API to process a payment. Your analytics dashboard calls an API to query events.

Think of an API as a menu. You don't need to know how the kitchen works — you just need to know what you can order, how to order it, and what you'll get back.

REST Principles

REST (Representational State Transfer) is an architectural style for designing networked APIs. It's not a protocol — it's a set of constraints that, when followed, produce APIs that are predictable, scalable, and easy to work with.

  • Stateless: every request contains all the information needed to process it. The server holds no session state between requests.
  • Uniform interface: consistent conventions for resources, verbs, and responses — regardless of what the resource is.
  • Client-server separation: the client and server evolve independently as long as the interface contract is upheld.
  • Cacheable: responses should indicate whether they can be cached, enabling performance optimisations.

REST URL Conventions

  • Use nouns, not verbs. /users not /getUsers. The HTTP verb provides the action.
  • Use plural resource names. /orders not /order.
  • Nest related resources. /users/123/orders for orders belonging to user 123.
  • Keep nesting shallow. More than two levels deep (/a/1/b/2/c) becomes hard to maintain.
  • Use kebab-case for multi-word paths. /user-profiles not /userProfiles.
  • Version your API in the path. /api/v1/users — makes breaking changes safe to ship.

Example: User + Orders API

GET    /api/v1/users           # list all users
POST   /api/v1/users           # create a user
GET    /api/v1/users/123       # get user 123
PATCH  /api/v1/users/123       # update user 123
DELETE /api/v1/users/123       # delete user 123

GET    /api/v1/users/123/orders      # list orders for user 123
POST   /api/v1/users/123/orders      # create order for user 123
GET    /api/v1/users/123/orders/456  # get order 456 of user 123

GraphQL vs REST

GraphQL is an alternative query language for APIs, developed by Facebook (Meta). Instead of multiple fixed endpoints, there's a single endpoint where clients specify exactly what data they need.

DimensionRESTGraphQL
Data fetchingFixed response shape per endpointClient specifies exact fields needed
Over-fetchingCommon — endpoints return full objectsEliminated — get only what you ask for
Under-fetchingCommon — multiple requests to assemble dataEliminated — fetch related data in one query
CachingSimple — HTTP caching works naturallyComplex — requires specialised caching (persisted queries)
Learning curveLow — widely understood, great toolingHigher — schema design, resolvers, N+1 problems
Best forPublic APIs, simple CRUD, microservicesComplex product APIs, mobile apps, many different clients
Rule of thumb: Start with REST. Move to GraphQL if your clients have highly variable data needs (e.g., a mobile app and a web app needing very different shapes of data from the same backend).

API Versioning

Once an API is public (used by external clients, mobile apps, or partners), changes that break the contract are extremely costly. Versioning lets you evolve the API without breaking existing consumers.

URL Versioning (recommended)

  • /api/v1/users
  • Explicit and visible
  • Easy to test and cache
  • Most common in practice

Header Versioning

  • Accept: application/vnd.api+json;version=2
  • Keeps URLs clean
  • Harder to test manually
  • Used by GitHub, Stripe

Backward-compatible changes (adding new fields, new endpoints, new optional parameters) don't need a version bump. Breaking changes (removing fields, changing types, changing behaviour) require a new version.

Maintain at least one previous major version with a sunset date. Give consumers 6–12 months to migrate.

API Authentication

MethodHow It WorksBest For
API Keys A static secret token sent in headers (Authorization: Bearer sk_live_... or X-API-Key: ...) Server-to-server integrations, simple public API access. Easy to implement; rotate if leaked.
OAuth 2.0 Delegated authorisation protocol. User grants an app permission to act on their behalf without sharing their password. Issues short-lived access tokens + refresh tokens. User-facing apps needing access to third-party resources ("Login with Google", Stripe Connect). Industry standard.
JWT JSON Web Tokens — signed, self-contained tokens encoding user identity and claims. Verified without a database lookup. Stateless auth in microservices, mobile apps. Common output of OAuth flows. Verify the signature; don't trust unverified JWTs.

For PMs: Speccing API Work

What to Include in API Requirements

  • Consumer context: who calls this endpoint — mobile app, web frontend, partner integration? Their needs should shape the response shape.
  • Resource model: what data is being created, read, updated, or deleted? What fields are required vs optional?
  • Error cases: what should happen when inputs are invalid, the resource doesn't exist, or the user lacks permission? Enumerate the expected error codes and messages.
  • Rate limits and pagination: for list endpoints, how many records? What's the pagination strategy (cursor, offset)?
  • Backward compatibility commitment: is this a public API? Who else might depend on it?

Backward Compatibility

Changing an API field name, removing a field, or changing a field type is a breaking change. If mobile apps consume this API, a breaking change could break old app versions that users haven't updated. Always ask: "Who else depends on this, and what will break for them?"

API Contracts as a Communication Tool

Agreeing on the API contract (request/response schema) upfront lets frontend and backend teams work in parallel. Tools like OpenAPI / Swagger let you define the contract as a spec, auto-generate docs, and mock the API so the frontend team doesn't have to wait for backend implementation.
