Code Review

How to give feedback that improves the code without damaging the relationship, and how to receive it without getting defensive.

What code review is actually for

Code review exists to catch bugs, share knowledge, and maintain consistency. That's it. It's not a quality gate that proves work is done. It's not a place to demonstrate your seniority. It's not an opportunity to rewrite something in your preferred style.

The most common failure mode is treating review as a blocking checkpoint rather than a collaborative conversation. When reviews slow things down by days, people start avoiding them — submitting large PRs to reduce the number of review cycles, or merging without full engagement. Both make things worse.

Before you submit a PR

Review your own diff first. Read every line as if someone else wrote it. This sounds obvious and most people skip it. You'll catch at least one thing you want to change before anyone else sees it.

Write a description that explains what changed and why — not just a link to the ticket. If there are decisions you made that aren't obvious from the code, say so. If there are areas you're uncertain about and want specific feedback on, point to them. A PR that arrives with context is reviewed faster and more usefully than one that arrives bare.

Keep PRs small. A 50-line change gets reviewed carefully. A 1,500-line change gets rubber-stamped, or gets stuck in review for four days because nobody has the time and headspace to engage with it properly. If the feature is big, find the seam where you can split it.

Giving feedback well

The most important thing is to separate observations about the code from opinions about the person. "This function is doing three things" is a useful observation. "Why would you write it like this" is not.

Prefix comments with their weight. A nit is something small and stylistic that you could take or leave. A blocker is something that must change before merge. Reviewers who mark everything as a blocker teach authors to tune them out. Reviewers who never distinguish between nits and blockers leave authors guessing what they actually need to fix.

Comment prefixes that help nit: — trivial style preference, take it or leave it
suggestion: — worth considering but not blocking
question: — I don't understand this, explain or change it
blocker: — this must change before merge
praise: — genuinely good, say so

If you'd write the code differently, consider whether your way is objectively better, merely different, or just more familiar. Style preferences that aren't covered by an agreed linting rule aren't worth raising. If the style argument comes up repeatedly, the answer is to add it to the linter, not to keep having the same comment thread.

Receiving feedback

Your first reaction to critical feedback is almost always worse than your second reaction. Sleeping on a frustrating review comment before responding is not weakness — it's just accurate self-knowledge about how humans work.

If a comment is unclear, ask for clarification before defending your approach. "Can you say more about what you're concerned about?" is better than "I did it this way because..." The reviewer might have a valid concern they expressed poorly, and you won't find out if you go straight to defending.

If you disagree and you've thought about it, say so directly with your reasoning. "I see the concern but I think X because Y — happy to discuss" is a legitimate response. Silently ignoring feedback and merging is not.

Async vs synchronous review

Most review happens asynchronously. This works well for small, clear changes. It breaks down for anything architecturally significant or contentious — comment threads are a poor substitute for a 20-minute conversation.

If a PR has generated more than six comment threads or been open for more than three days without resolution, schedule a call. What feels like a disagreement about implementation is usually a disagreement about requirements or constraints that isn't surfaced in the code.

Good reviewer habits

  • Review within one business day of request
  • Label comment weight (nit/suggestion/blocker)
  • Explain the why, not just the what
  • Acknowledge things done well
  • Suggest, don't demand on style questions

Review antipatterns

  • Reviewing 1,500-line PRs by scrolling and approving
  • Style nitpicks without a shared style guide
  • "This should be refactored" with no specifics
  • Blocking on personal preference disguised as principle
  • Leaving review open for days without explanation

When to escalate

If you and the author genuinely disagree on something significant and neither is convinced, bring in a third person. Don't let a PR block indefinitely because two people are stuck. Choose someone who has context, ask them to weigh in, and commit to accepting their input.
