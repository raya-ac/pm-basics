Security Basics

What every PM should understand about keeping products and data secure.

Why Security Is a PM Concern

Security isn't just an infrastructure problem. Every feature you ship can introduce a vulnerability. PMs who don't think about security ship features that expose user data, open attack surfaces, and erode trust.

Features Ship Vulnerabilities

New file upload? New attack vector. New user input field? Potential injection point. Every feature expands the attack surface.

Trust Is a Product Attribute

Users don't separate "the product" from "security." A breach doesn't just affect infra — it damages the brand permanently.

Compliance Is a Blocker

Enterprise sales often require SOC 2, ISO 27001, or HIPAA. Security work is a revenue enabler, not just cost.

OWASP Top 10

The OWASP Top 10 is the industry standard list of the most critical web application security risks.

1
Injection
SQL, NoSQL, OS injection — untrusted data sent to an interpreter. Use parameterised queries, never string concat.
2
Broken Authentication
Weak passwords, missing MFA, session fixation. Use battle-tested auth libraries (Auth0, Cognito) rather than rolling your own.
3
Sensitive Data Exposure
Storing PII or secrets unencrypted. Encrypt at rest and in transit. Don't log sensitive fields.
4
Broken Access Control
Users accessing resources they shouldn't. Enforce server-side, not just client-side. Test every endpoint with a non-privileged user.
5
Security Misconfiguration
Default credentials, open S3 buckets, debug mode in prod, verbose error messages. Infrastructure as Code helps enforce correct config.
6
Cross-Site Scripting (XSS)
Injecting scripts into pages viewed by other users. Escape all user-generated content. Use Content-Security-Policy headers.
7
Insecure Deserialization
Deserialising untrusted data can lead to remote code execution. Validate and sanitise all incoming data structures.
8
Using Vulnerable Components
Outdated libraries with known CVEs. Use Dependabot, Snyk, or similar to automate dependency updates.
9
Insufficient Logging
No audit trail means breaches go undetected. Log auth events, access to sensitive data, and errors. Set up alerts.
10
Server-Side Request Forgery
Server fetches a URL supplied by an attacker, potentially reaching internal services. Validate and whitelist URLs.

Authentication vs Authorization

Authentication (AuthN)

Who are you? — Verifying identity via passwords, MFA, OAuth, SSO. Most apps use a third-party provider (Auth0, Firebase Auth, Cognito).

Authorization (AuthZ)

What can you do? — Permissions and roles after identity is confirmed. Role-based (RBAC) or attribute-based (ABAC) access control.

Common mistake: Checking auth only on the frontend. Always enforce authorization server-side on every request.

Secure SDLC

Security integrated into every phase of development — not bolted on at the end.

Design

Threat modeling — "what could go wrong?" before writing code. Define trust boundaries and data flows.

Development

SAST tools (Snyk, SonarQube) in CI. Peer review with security lens. No hardcoded secrets.

Pre-Launch

Penetration testing for high-risk features. DAST scanning (OWASP ZAP). Dependency audit.

Post-Launch

Bug bounty program. Continuous monitoring. Regular security reviews. Incident response plan.

Compliance Basics

GDPR

EU data privacy law. Key requirements: lawful basis for processing, right to erasure, data minimisation, breach notification within 72h.

SOC 2

Security audit framework. Type I = point-in-time, Type II = over 6–12 months. Enterprise customers often require it.

PCI-DSS

Required if you handle card payments. Use Stripe/Braintree to avoid direct card handling — they're PCI compliant so you don't have to be.

HIPAA

US health data law. Applies if you handle PHI (Protected Health Information). Requires BAA with all vendors who touch the data.

For PMs: Security in Practice

Write security requirements explicitly. "Users can only access their own data" is an acceptance criterion, not an assumption.
Prioritise security bugs as P0/P1 if they expose user data or allow privilege escalation. Don't let them sit in the backlog.
When communicating incidents: be fast, be specific about what was affected, explain what you've done to fix it and prevent recurrence.
Involve security early in design. A 30-minute threat model session in discovery is cheaper than a pen test finding in production.
