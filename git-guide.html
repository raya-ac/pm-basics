Git Guide

Branches, rebases, conflicts, and the history decisions that feel irrelevant until six months later when you're trying to bisect a regression.

The mental model that actually helps

Git stores snapshots, not diffs. Each commit is a complete picture of your repository at that point in time, with a pointer to its parent. Branches are just named pointers to commits. Once this clicks, a lot of the confusing commands start making sense.

The three trees you need to know: the working directory (what's on disk), the index/staging area (what you've git added), and HEAD (the last commit). Most commands move data between these three places.

Branching strategies

There are a few common patterns, and teams argue about them more than the differences warrant.

Trunk-based development

Everyone commits to main (or a short-lived branch that merges within a day). You ship to production frequently. Feature flags control what users see, not branches. This is what Google, Facebook, and most high-performing teams do. It requires good automated testing and a mature deployment pipeline.

Gitflow

Long-lived develop and release branches, with feature branches off develop. Made sense for software with explicit release cycles (mobile apps, packaged software). For continuous web deployment it tends to create merge hell. A lot of teams adopted it in ~2012 and haven't questioned it since.

GitHub Flow

Feature branches off main, PR, merge, deploy. Simpler than Gitflow, works well for teams deploying frequently. The main risk is a long-lived feature branch that diverges significantly before merging.

The honest take Trunk-based development is harder to start with and significantly better in practice. If your team has good test coverage and can deploy safely, the overhead of managing multiple long-lived branches is pure cost with no benefit.

Commits worth making

A good commit is a logical unit of change with a message that explains why, not just what. The diff shows what changed; the message needs to explain the reasoning.

The format that ages well: a short subject line (under 72 characters), a blank line, then a body explaining the problem this commit solves and any relevant context. Referencing an issue or ticket number in the body is useful when the project has them.

fix: prevent race condition in cart checkout

When two tab sessions submitted the order form simultaneously,
both would pass the inventory check before either decremented
the stock count. Added a pessimistic lock on the order creation
transaction.

Commits like wip, asdf, or fix fix fix are fine during active development — just squash them before the PR lands on main.

Rebase vs merge

Both integrate changes from one branch into another. They produce different histories.

Merge creates a merge commit that shows two branches converged. The history is accurate but can become noisy on busy repos. Rebase replays your commits on top of the target branch as if you'd branched from the latest commit. The history is linear and clean, but you've rewritten commits — which causes problems if others have based work on the original commits.

The rule most teams settle on: rebase to clean up your own branch before a PR, never rebase shared branches.

Don't do this git push --force on a branch someone else is working on. Force-pushing rewrites history on the remote, and anyone who has pulled that branch now has a diverged history. Use git push --force-with-lease if you must force-push — it at least checks that nobody else has pushed since your last fetch.

Fixing mistakes

The most useful recovery commands, in order of how drastic they are:

  • git commit --amend — fix the last commit message or add a forgotten file
  • git revert <hash> — creates a new commit that undoes a previous one. Safe on shared branches.
  • git reset --soft HEAD~1 — undoes the last commit but keeps the changes staged
  • git reset --hard HEAD~1 — undoes the last commit and discards the changes. Hard to recover from.
  • git reflog — your safety net. Shows every position HEAD has been in for the last 90 days. If you've lost commits, they're probably still here.

Merge conflicts

A merge conflict happens when two branches have modified the same lines of the same file. Git can't decide which version to keep, so it marks the file and asks you to decide.

The conflict markers show you three things: your changes (HEAD), the incoming changes, and the separator. Your job is to edit the file to reflect what the combined result should be — which sometimes means picking one side, sometimes means writing something new that incorporates both.

<<<<<<< HEAD
const timeout = 3000;
=======
const timeout = 5000;
>>>>>>> feature/increase-timeout

After resolving: git add the file, then git merge --continue or git commit. Don't leave conflict markers in the code. They will end up in production eventually.

Good git habits

  • Short-lived branches (days, not weeks)
  • Descriptive commit messages with context
  • Squash or fixup before PR merge
  • Pull and rebase before pushing
  • Tag releases so you can find them later

Problems in the making

  • Committing secrets or credentials (use git-secrets)
  • Binary files in the repo (use Git LFS or a CDN)
  • Force-pushing to main
  • Branch names like test2-final-v3
  • 500MB node_modules committed to git
