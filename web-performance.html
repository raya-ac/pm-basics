Web Performance

Why speed matters, how to measure it, and how to fix it.

Why Performance Matters

Performance is a product feature. Slow pages cost money, lose users, and hurt rankings. The data is unambiguous:

  • Amazon found every 100ms of latency cost them 1% in sales.
  • Google reports that a 1-second delay in mobile load times can reduce conversions by up to 20%.
  • 53% of mobile users abandon a site that takes longer than 3 seconds to load (Google/SOASTA research).
  • Walmart saw a 2% increase in conversions for every 1 second of improvement in load time.

SEO Impact

Google uses Core Web Vitals as a direct ranking signal since 2021. A slow site doesn't just lose users — it also appears lower in search results, compounding the traffic loss.

User Perception

Users perceive responses under 100ms as instant. At 1s, flow is interrupted. At 10s, users abandon. Performance directly shapes whether users feel your product is trustworthy and professional.

Core Web Vitals

Google's Core Web Vitals are three user-centric metrics that measure real-world page experience. They're tracked in Google Search Console and used as ranking signals.

How to Measure Performance

ToolTypeBest For
LighthouseSynthetic (lab)Quick audits in Chrome DevTools or CI pipelines. Scores 0–100 across Performance, A11y, SEO.
PageSpeed InsightsLab + FieldGoogle's free tool. Shows lab data alongside real-user CWV data from the Chrome UX Report (CrUX).
WebPageTestSyntheticAdvanced waterfall analysis, filmstrips, multi-region testing. Shows exactly what loads when.
Chrome DevToolsLabNetwork tab, Performance tab, Coverage tab — for deep debugging of specific pages.
RUM (Real User Monitoring)FieldTools like Datadog, Sentry, or web-vitals.js — captures real user experience across all devices.
Lab vs Field data: Lab data (Lighthouse) is reproducible but artificial. Field data (RUM/CrUX) reflects what real users experience. Use both — lab for debugging, field for tracking trends.

Common Performance Fixes

Image Optimisation

  • Use WebP or AVIF format — typically 25–50% smaller than JPEG/PNG at equivalent quality.
  • Add loading="lazy" to images below the fold so they don't block initial load.
  • Always specify width and height attributes to prevent layout shift (CLS).
  • Use responsive images with srcset so mobile devices don't download desktop-sized images.
<img src="hero.webp" width="800" height="450"
     loading="lazy" alt="Dashboard screenshot"
     srcset="hero-400.webp 400w, hero-800.webp 800w"
     sizes="(max-width: 600px) 400px, 800px">

JavaScript Bundle Size

  • Code splitting: load only the JS needed for the current route, not the whole app.
  • Tree shaking: remove unused exports at build time (Webpack, Rollup, esbuild do this automatically).
  • Dynamic imports: defer loading of non-critical modules with import().
  • Audit with the Chrome Coverage tab — often 40–60% of JS is never executed on a given page.

Caching

  • Set long Cache-Control: max-age headers for versioned static assets (CSS, JS, images).
  • Use a CDN (Cloudflare, Fastly, CloudFront) to serve assets from edge nodes close to users.
  • Use ETag or Last-Modified headers so browsers can validate stale cache entries cheaply.

Render-Blocking Resources

  • Move non-critical CSS to load asynchronously. Inline critical CSS directly in the <head>.
  • Add defer or async to script tags that don't need to run before page render.
  • Use <link rel="preload"> to tell the browser to fetch critical fonts or images early.

Server Response Time (TTFB)

  • Target Time to First Byte (TTFB) under 800ms. Over 1.8s is considered poor.
  • Use server-side caching (Redis, Varnish) to avoid recomputing responses.
  • Review slow database queries — N+1 problems are a common culprit.
  • Consider edge compute (Cloudflare Workers, Vercel Edge Functions) for global TTFB reduction.

Performance Budget

A performance budget is a set of limits your team agrees not to exceed — for example, total page weight under 300KB, LCP under 2.5s, or Lighthouse performance score above 85.

Budgets are most powerful when enforced in CI. Tools like Lighthouse CI, Bundlesize, and Size-Limit will fail a build if a PR pushes the bundle over the agreed threshold.

// .lighthouserc.js
module.exports = {
  assert: {
    assertions: {
      'categories:performance': ['error', { minScore: 0.85 }],
      'largest-contentful-paint': ['warn', { maxNumericValue: 2500 }],
      'cumulative-layout-shift': ['error', { maxNumericValue: 0.1 }],
    },
  },
};

A budget forces the conversation to happen before a feature ships, not after a customer complains.

For PMs: Prioritising Performance Work

How to Frame It

Performance is often invisible until it's bad. Frame perf work in business terms: conversion rate, bounce rate, SEO ranking, and support ticket volume. "LCP went from 4.2s to 1.9s" is less compelling to stakeholders than "we expect a 5% uplift in trial signups based on industry benchmarks for this load time improvement."

Fitting Into OKRs

  • Objective: Improve the quality and reliability of the core product experience.
  • KR: Achieve a Lighthouse performance score ≥ 85 across all key pages by Q3.
  • KR: Reduce LCP on the landing page from 3.8s to under 2.5s.
  • KR: Reduce mobile bounce rate by 10% through performance improvements.

When to Prioritise

  • If Core Web Vitals are in the red in Google Search Console — SEO risk is immediate.
  • Before a major marketing push or launch — you want the funnel to convert.
  • When growth is plateauing — perf improvements can unlock conversion gains without new features.
  • When your largest customers operate in bandwidth-constrained regions.
Rule of thumb: performance improvements have compounding returns. A 1s improvement today saves engineering time tomorrow because less performance debt accumulates.
