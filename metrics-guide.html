Metrics Guide

How to pick the right metrics, avoid vanity traps, and connect numbers to decisions that actually matter.

⭐ North Star Metric

Definition: The single metric that best captures the core value your product delivers to users. It aligns the entire company on what "success" means. Everything else is either a leading indicator of it or a guardrail protecting it.
Weekly active editors
Figma's North Star
Nights booked
Airbnb's North Star
Messages sent per day
Slack's North Star

How to define yours

Ask: "What action, when a user does it repeatedly, most reliably predicts long-term retention and revenue?" It should be: specific (not "engagement"), measurable (not "happiness"), and action-based (a thing users do, not a thing you feel).

📈 Input vs Output Metrics

Output metrics (lag)

Measure the result. Hard to influence directly in the short term. Tell you if you succeeded — after the fact.

  • Monthly recurring revenue (MRR)
  • Net Promoter Score (NPS)
  • Churn rate
  • Daily active users (DAU)

Input metrics (lead)

Measure the behaviors that drive the output. Actionable and faster-moving. Ship work that moves these; output will follow.

  • Onboarding completion rate
  • Feature adoption rate
  • Time to first value
  • Support ticket volume
PM tip: Manage to input metrics, report on output metrics. Your team can directly move inputs; outputs are the lagged result of many inputs across many teams.

💚 HEART Framework (Google)

A structured way to define user-centered metrics for any product or feature. For each dimension, define Goal → Signal → Metric.

H
Happiness
User satisfaction and sentiment. NPS, CSAT, app store rating.
E
Engagement
Depth and frequency of use. Sessions/week, features used per session.
A
Adoption
New users reaching key actions. % of new signups completing setup.
R
Retention
Users returning over time. Day-7 retention, monthly active rate.
T
Task Success
Efficiency and completion of specific tasks. Time on task, error rate, completion rate.
Usage: Pick 1–2 HEART dimensions most relevant to the feature you're measuring. Defining all 5 for every feature creates metric overload.

🎯 Setting Good Feature Success Metrics

The success metric formula

A good feature metric is: specific (one number, not "engagement"), time-bound (measured at 30 days post-launch), directional (higher is better), and connected to the North Star.

Template: "[metric name] will increase from [baseline] to [target] within [timeframe] of launch, as measured by [data source]."

Always define a guardrail metric

The metric you're not willing to hurt in order to move the primary metric. Example: increasing checkout conversion rate should not increase payment fraud rate. Guardrails prevent local optimization that damages the broader system.

🔧 Instrumentation Checklist

Before you build

  • Define success metric and baseline
  • Identify what events need to be tracked
  • Agree on event naming convention with eng
  • Confirm tracking is in the acceptance criteria
  • Verify analytics pipeline can handle the volume

At launch

  • Verify events are firing in staging before prod
  • Check event schema matches what was planned
  • Confirm dashboard is populated with real data
  • Set up alerts for anomalous metric movement
  • Document event definitions in the data dictionary
