Technical Documentation

The types of technical docs engineers write, how they're structured, and how to keep them useful over time.

📄 Types of Technical Docs

README

Entry point for any repo or project. Answers: what is this, how do I run it, how do I contribute. Lives in the repo root. Goes stale fastest — keep it short and accurate.

Always present

Architecture Decision Record (ADR)

Documents why a significant technical decision was made, what alternatives were considered, and what trade-offs were accepted. The org's institutional memory for "why does it work this way?"

Decision log

Runbook

Step-by-step operational guide for a specific task or incident scenario. Written for an on-call engineer at 3am. Should be executable without deep context. "If X happens, do Y."

Ops critical

API Documentation

Describes every endpoint: method, URL, request params, request body, response schema, error codes, authentication, rate limits. Often generated from OpenAPI/Swagger specs.

External-facing

Design Doc (RFC)

Proposes a significant change before any code is written. Captures problem statement, proposed solution, alternatives, open questions. Reviewed async; builds alignment without meetings.

Pre-work

Onboarding Guide

Helps a new team member become productive in days, not weeks. Covers dev environment setup, key repos, team processes, who owns what, where to ask questions.

People-facing

📚 README Anatomy

# Project Name
One-sentence description of what this does and why it exists.

## Quick Start
```bash
git clone https://github.com/org/repo
cd repo
npm install
npm run dev   # starts dev server at http://localhost:3000
```

## What It Does
2-3 paragraphs or bullet points. No jargon. Link to deeper docs.

## Architecture Overview
Optional diagram or link to design doc. Where does this fit in the system?

## Configuration
Environment variables, config files, secrets needed. Reference .env.example.

## Running Tests
```bash
npm test          # unit tests
npm run test:e2e  # end-to-end tests
```

## Deployment
Link to runbook or CI/CD pipeline. Who approves deploys?

## Contributing
Branch naming, PR process, code review expectations.

## Owners
Team name + Slack channel. On-call rotation link.
PM tip: Check the README health of repos your team owns. A stale README is a sign of poor docs culture. Ask engineers: "If someone joined today, could they ship using this README alone?"

📋 ADR Format

When to write an ADR: Any decision that is hard to reverse, affects multiple teams, involves a significant trade-off, or will confuse future engineers who weren't in the room. 
# ADR-0023: Use PostgreSQL for primary data store

## Status
Accepted | 2024-02-14

## Context
We need a primary data store for the new billing service. The team
has experience with both MySQL and PostgreSQL. We need ACID
compliance, JSON support, and good support for complex queries.

## Decision
We will use PostgreSQL 16 hosted on AWS RDS.

## Alternatives Considered
- **MySQL**: Familiar, but inferior JSON support and window functions.
- **DynamoDB**: Better at scale, but our query patterns are relational.
- **CockroachDB**: Distributed, but operational complexity not justified yet.

## Consequences
**Positive:**
- Full ACID compliance for financial data
- Native JSONB columns for flexible metadata
- Team has deep existing expertise

**Negative:**
- Vertical scaling has limits (mitigated by read replicas)
- RDS cost higher than self-managed at scale

🔧 Runbook Structure

Header Block

Title: "Restart the payment processor service"
Severity: SEV-2
Owner: payments team
Last verified: 2024-03-01
Alert link: PagerDuty alert #1234

When to use this runbook

Describe the exact symptoms that trigger this runbook. Include alert names, error messages, dashboard links. Engineers must know immediately whether this is the right doc.

Steps (numbered, atomic)

Each step is a single action with the exact command or UI action. Include expected output. Flag steps that require elevated permissions. Add "if this fails, see Step X."

Verification

How do you know it worked? Specific metric to check, specific API call to make, specific log line to see. Do not end a runbook without a success criterion.

Escalation path

If the runbook doesn't resolve it: who to page, what channel to post in, what context to include in the escalation. Never leave an engineer stranded.

Rollback

For any runbook involving a change: how to undo every step. Especially critical for config changes and database migrations.

🔄 Docs-as-Code Workflow

Author
Write docs in Markdown, co-located with code
Docs live in the same repo as the code they describe. ADRs in /docs/adr/, runbooks in /docs/runbooks/, API specs as openapi.yaml. Version-controlled alongside code.
Review
Docs reviewed in the same PR as the code change
If a PR changes an API, it must update the API docs. If it changes ops behavior, it must update the runbook. CI can lint docs for broken links and required sections.
Publish
Auto-publish on merge
CI/CD pipeline deploys docs site (Docusaurus, MkDocs, etc.) on every merge to main. No manual uploads. Docs site always reflects latest merged state.
Maintain
Audit and retire stale docs quarterly
Schedule a quarterly "doc debt" review. Archive docs for deprecated systems. Update "last verified" dates. Delete duplicates. Stale docs are worse than no docs — they mislead.

⚠ Common Mistakes

MistakeWhy it's harmfulFix
Writing docs after the factContext is lost; authors don't remember the alternatives consideredRequire docs in the PR, not as a follow-up ticket
Docs in a wiki, code in a repoDocs drift from code reality within weeksCo-locate docs with code
No owner listedNobody updates it; becomes stale silentlyEvery doc has a named team or Slack channel
Explaining the "what", not the "why"Engineers re-derive the same decisions; ADRs exist for this reasonDocument motivation and alternatives, not just outcome
Too long, no structureNobody reads wall-of-text docsUse headers, bullets, code blocks; keep each section short
No "last verified" date on runbooksEngineer follows 2-year-old steps on a system that's been rebuiltRequire date + name on all operational docs

✍ Writing Tips for Engineers

Write for your reader's context

A README is for a new hire. A runbook is for an on-call engineer under stress. An ADR is for a future engineer wondering why. Adapt vocabulary and depth to who will read it and when.

Lead with the answer

Start with what the reader needs most. "This service handles payment retries" beats a 3-paragraph history of why the service exists. Put context after conclusions.

Use active voice for instructions

"Run make build" not "The build can be started by running make build." Passive voice adds friction. Instructions should be imperative and direct.

Test your docs

Ask a teammate to follow the doc with no help from you. Where they get stuck is where the doc fails. Treat this like a usability test — every confusion is a gap to fix.
