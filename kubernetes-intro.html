Container orchestration -- how teams run Docker at scale across many servers.

What is Kubernetes?

Kubernetes (abbreviated k8s -- the 8 letters between K and s) is an open-source system for automating the deployment, scaling, and management of containerised applications. Originally created at Google, it is now maintained by the Cloud Native Computing Foundation (CNCF) and is the industry standard for running containers in production.

If Docker is the technology that packages an application into a container, Kubernetes is the system that decides where that container runs, keeps it healthy, scales it up when traffic spikes, and rolls out new versions without downtime.

Kubernetes handles four hard problems at scale:

  • Scheduling: Placing containers onto the right servers based on available CPU and memory.
  • Self-healing: Automatically restarting crashed containers, replacing unhealthy ones, and rescheduling when a server fails.
  • Scaling: Adding or removing container instances in response to traffic load -- automatically or on command.
  • Rolling deploys: Updating containers to a new version gradually, with zero downtime and automatic rollback on failure.

Core Concepts

Pod
The smallest deployable unit in k8s. A pod wraps one or more containers that share a network and storage. Most pods contain a single container.
Node
A physical or virtual machine in the cluster. Kubernetes schedules pods onto nodes based on available resources. A cluster typically has many nodes.
Cluster
The full set of nodes managed by Kubernetes. The control plane manages the cluster state; worker nodes run the actual application pods.
Deployment
Describes the desired state of your application: which image to run, how many replicas, update strategy. Kubernetes reconciles actual state with the desired state continuously.
Service
A stable network endpoint that routes traffic to a set of pods. Since pods are ephemeral and change IPs, a Service provides a consistent address for other services to connect to.
Ingress
Manages external HTTP/HTTPS traffic into the cluster. Routes requests from the internet to the correct internal Service based on hostname or URL path rules.
Namespace
A virtual partition within a cluster. Teams use namespaces to separate environments (staging vs production) or different services within a cluster.
ConfigMap
Stores non-secret configuration data (feature flags, API endpoints, env vars) separately from the container image, so config can change without rebuilding the image.

How a Deploy Works in Kubernetes

From a code push to running in production, here is the typical flow:

1

Code push triggers CI

Engineer merges a pull request. The CI pipeline runs tests and, on success, builds a new Docker image tagged with the commit SHA.

2

Image pushed to registry

The new image is pushed to a container registry (AWS ECR, GCR, etc.) with its unique tag. The old image remains available for rollbacks.

3

Deployment manifest updated

The CI pipeline updates the Kubernetes Deployment manifest with the new image tag, then applies it to the cluster with kubectl apply -f.

4

Kubernetes rolls out new pods

k8s starts new pods with the updated image. By default, it uses a rolling update: brings up new pods one at a time before terminating old ones. Traffic shifts gradually.

5

Health checks gate the rollout

k8s runs readiness probes on each new pod. If a pod fails its health check, the rollout pauses automatically -- old pods keep serving traffic while engineers investigate.

6

Old pods terminate

Once all new pods are healthy and serving traffic, the old pods are gracefully terminated. A full rollback is one command: kubectl rollout undo deployment/my-app.

Key kubectl Commands

kubectl is the command-line tool for interacting with a Kubernetes cluster. Engineers use these in day-to-day operations and incident response:

# List all pods in the current namespace
kubectl get pods

# List pods in all namespaces
kubectl get pods --all-namespaces

# Get detailed info about a specific pod (useful for debugging)
kubectl describe pod <pod-name>

# Stream logs from a pod
kubectl logs -f <pod-name>

# Apply a configuration file (deploy a new version, update config)
kubectl apply -f deployment.yaml

# Check the status of a rolling deployment
kubectl rollout status deployment/my-app

# Roll back to the previous version
kubectl rollout undo deployment/my-app

# Scale a deployment to 5 replicas manually
kubectl scale deployment/my-app --replicas=5

# Open a shell inside a running pod
kubectl exec -it <pod-name> -- sh

Scaling in Kubernetes

Horizontal Pod Autoscaler (HPA)

Automatically adjusts the number of pod replicas based on observed CPU utilisation or custom metrics (like request rate). When traffic spikes, k8s adds pods. When it drops, pods are removed.

Resource Requests and Limits

Each container declares how much CPU and memory it requests (guaranteed) and its limit (maximum). This lets the scheduler pack pods efficiently and prevents a runaway process from starving its neighbours.

Setting accurate resource requests is one of the most impactful reliability improvements a team can make. Under-provisioned limits cause OOM kills; over-provisioned requests waste money on idle capacity.

For PMs: What Kubernetes Enables

What k8s enables for your product:

  • Zero-downtime deploys: Rolling updates mean users never hit a "down for maintenance" page. New versions roll out while old ones keep serving traffic.
  • Auto-scaling for traffic spikes: If you run a marketing campaign and traffic 10x's, k8s can spin up additional pods within seconds -- no manual intervention, no weekend on-call scramble.
  • Environment parity: Staging and production run identical Kubernetes configs, making staging a reliable signal of production behaviour.
  • Faster incident recovery: Self-healing means crashed pods restart automatically. Many incidents resolve themselves before users notice.

Questions to ask your SRE or platform team:

  • Do we have Horizontal Pod Autoscaling configured for high-traffic services? What metrics trigger scaling?
  • What is our rollback strategy for a bad deploy? How long does a rollback take?
  • Do our pods have readiness and liveness probes configured? (These are what make zero-downtime deploys and self-healing actually work.)
  • Are resource requests and limits set accurately? When did we last review them against actual usage?
  • What is our process for a full cluster incident vs. a single pod issue?
