Docker Basics

Containers explained -- what they are, why they matter, and what PMs need to know.

docker containers devops deployment

What is a Container?

A container is a lightweight, standalone, executable package of software that includes everything needed to run an application: code, runtime, system tools, system libraries, and settings. Containers isolate software from its environment and ensure it works uniformly across development, staging, and production.

The famous problem containers solve: "It works on my machine." When a developer says this, it usually means their local environment has different OS versions, library versions, or config than production. Docker eliminates that gap by packaging the environment alongside the code.

Virtual Machines (VMs)
  • Include a full OS guest layer
  • Slow to start (minutes)
  • Heavy -- GBs per VM
  • Strong isolation boundary
  • Good for running different OSes
Containers (Docker)
  • Share the host OS kernel
  • Fast to start (seconds or less)
  • Lightweight -- MBs per image
  • Process-level isolation
  • Ideal for microservices and CI/CD

Key Concepts

Image
A read-only template used to create containers. Built from a Dockerfile. Think of it as a snapshot of the application and its environment. Images are versioned and stored in registries.
Container
A running instance of an image. You can run many containers from the same image. Containers are ephemeral by default -- stopping one loses any data written inside it.
Dockerfile
A text file with instructions for building a Docker image. Defines the base OS, copies in code, runs install commands, and sets the startup command.
Registry
A storage and distribution system for Docker images. Docker Hub is the public default. Teams use private registries like AWS ECR, Google Artifact Registry, or GitHub Container Registry.
Volume
A way to persist data generated and used by containers. Volumes live outside the container filesystem so data survives container restarts and removals.
Network
Docker's virtual network layer. Containers on the same network can talk to each other by name. By default, containers are isolated from the host network.

Dockerfile Example

A Dockerfile for a Node.js application. Each line is an instruction that creates a layer in the image:

# Use the official Node 18 Alpine image as the base (small footprint)
FROM node:18-alpine

# Set the working directory inside the container
WORKDIR /app

# Copy dependency manifests first (enables Docker layer caching)
COPY package*.json ./

# Install production dependencies
RUN npm install --production

# Copy the rest of the application source code
COPY . .

# Expose the port the app listens on
EXPOSE 3000

# Define the command to run when the container starts
CMD ["node", "server.js"]

The order of instructions matters for build performance. Copying package.json before the source code means Docker only re-runs npm install when dependencies change -- not on every code edit.

Common Docker Commands

Engineers use these commands daily. Useful to recognise them in PR descriptions, CI logs, and runbooks:

# Build an image from the Dockerfile in the current directory
docker build -t my-app:latest .

# Run a container from an image, mapping port 3000
docker run -p 3000:3000 my-app:latest

# Run in detached (background) mode
docker run -d -p 3000:3000 my-app:latest

# List running containers
docker ps

# Stream logs from a running container
docker logs -f <container-id>

# Stop a running container
docker stop <container-id>

# Start all services defined in docker-compose.yml
docker-compose up

# Rebuild and start services
docker-compose up --build

# Execute a command inside a running container (e.g. open a shell)
docker exec -it <container-id> sh

Docker in the Development Workflow

1

Local Development Parity

Engineers run docker-compose up locally to spin up the full application stack -- app server, database, cache -- in seconds. No "it works on my machine" surprises because everyone runs the same image.

2

CI/CD Pipeline

On every pull request, the CI system builds a Docker image and runs tests inside it. If tests pass, the image is pushed to a registry tagged with the commit SHA. The exact image that passed tests is what gets deployed.

3

Staging Environments

Staging runs the same Docker image as production, just pointed at a staging database and config. This makes staging a reliable pre-production gate -- if it works in staging, it will work in prod.

4

Production Deployment

Orchestrators like Kubernetes pull the approved image from the registry and run it across multiple servers. Rollbacks are as simple as telling the orchestrator to run the previous image tag.

For PMs: What Docker Enables

Why "works on my machine" is solved:

  • Before containers, subtle differences between developer laptops, CI servers, and production caused bugs that were hard to reproduce and expensive to fix. Docker packages the entire environment, making every environment identical.
  • This is why you should care: environment inconsistencies inflate engineering estimates and post-launch incidents. Teams with good containerisation have fewer "works in staging, broken in prod" incidents.

What Docker enables for deployment speed:

  • Faster deploys: Spinning up a new container takes seconds vs. minutes for a VM. This makes frequent small deploys practical.
  • Reliable rollbacks: Rolling back means redeploying the previous container image -- a deterministic, fast operation rather than a risky manual process.
  • Feature flags + containers: You can run multiple versions of a service simultaneously by running different container image tags, enabling gradual rollouts and A/B testing at the infrastructure level.
  • Onboarding speed: New engineers can run the full product locally with a single command. This reduces "first PR" time from days to hours.

Questions to ask your team: Are your environments containerised and consistent? How long does it take to roll back a bad deploy? How long does local setup take for a new hire?
