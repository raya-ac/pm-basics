System Design

The building blocks of distributed systems — what they do, when to reach for them, and what the tradeoffs actually are rather than what the interview prep guides say.

Start with the problem, not the architecture

The most common system design mistake is picking components before understanding requirements. "We need Kafka" and "we need a microservices architecture" are solutions looking for problems. Start with: what does this system need to do, how many users, what's the read/write ratio, what does failure look like, what's the acceptable latency?

Most systems don't need a distributed architecture at all. A single well-tuned Postgres instance can handle millions of rows and thousands of concurrent connections. A monolith deployed to a couple of large servers handles more traffic than most companies will ever see. Complexity has a cost. Don't pay it unless you have to.

Load balancers

A load balancer distributes incoming requests across multiple server instances. This serves two purposes: horizontal scalability (add more instances to handle more load) and availability (if one instance dies, the others keep serving).

Layer 4 load balancers route at the TCP level — fast but no awareness of HTTP. Layer 7 load balancers understand HTTP — they can route based on URL path, set headers, handle SSL termination, and make smarter decisions. Most web applications use L7.

Session affinity (sticky sessions) routes the same user to the same server instance. Useful if your application stores session state locally, but it undermines the load distribution and complicates deployments. Better to store session state externally (Redis, a database) so any instance can handle any request.

Caching

A cache stores the result of an expensive operation so it can be served cheaply next time. Expensive usually means slow (database query, external API call) or CPU-intensive (rendered template, computed value).

Cache invalidation

The hard part. There are two main strategies: time-to-live (the cache entry expires after N seconds) and explicit invalidation (you delete or update the cache entry when the underlying data changes). TTL is simpler but means users can see stale data. Explicit invalidation is accurate but easy to get wrong — if you forget to invalidate a cache somewhere, you ship bugs that are very hard to reproduce.

Where to cache

Browser cache, CDN, reverse proxy (Nginx/Varnish), application cache (Redis/Memcached), database query cache. Each sits at a different point in the stack and serves different purposes. The closer to the user, the faster — but also the harder to invalidate.

Cache stampede When a popular cache entry expires, many requests hit the database simultaneously to regenerate it. This can take down a database that was otherwise handling load fine. Mitigations: probabilistic early expiry, cache locks, or background regeneration before expiry.

Databases

Relational (PostgreSQL, MySQL)

Structured data, ACID transactions, powerful query language. The right choice for most applications. People reach for NoSQL prematurely. If your data has relationships and your queries need consistency, use a relational database.

Document stores (MongoDB, Firestore)

Flexible schema, good for hierarchical data where the shape varies. The "no schema" pitch sounds appealing until you've spent three hours writing a migration for data that was written in five different shapes over four years.

Key-value stores (Redis, DynamoDB)

Extremely fast for simple lookups. Redis in particular is excellent for caching, sessions, queues, and rate limiting. Not designed for complex queries.

Read replicas and sharding

Read replicas copy your primary database and serve read traffic. Cheap, effective, low complexity. Sharding splits your data across multiple database nodes by some key. Dramatically more complex. Use read replicas first, exhaust them, then consider sharding — most applications never need it.

Message queues

A queue decouples the producer of work from the consumer. Instead of processing something synchronously in the request path, you put it on a queue and a worker picks it up later.

Good for: sending emails, processing images, anything that can be retried, anything that takes more than a few hundred milliseconds. Not good for: things that need an immediate response to the user, things where ordering guarantees are critical and you haven't modelled that carefully.

RabbitMQ is a traditional message broker. Kafka is a distributed log — good for high-throughput event streaming where you want consumers to replay history. SQS is AWS's managed queue. Pick the simplest one that meets your requirements.

CAP theorem (the simplified version)

In a distributed system, during a network partition you must choose between consistency (all nodes see the same data) and availability (the system keeps responding). You can't have both when nodes can't communicate.

The practical implication: know whether your system needs to be CP (consistent, possibly unavailable during partitions) or AP (available, possibly returning stale data). Most user-facing web applications are fine with AP. Banking systems are not.

The honest take on system design interviews System design interviews test whether you can have a structured conversation about tradeoffs, not whether you know the right answer. There usually isn't one. Say what you're optimising for, explain why you'd pick one approach over another, and acknowledge the downsides. Interviewers who want you to recite a specific architecture are checking the wrong thing.
