CI/CD

Continuous integration and delivery: what the pipeline actually does, how to design one that doesn't make your team hate deployments, and why your deploy process is a symptom of your team culture.

CI vs CD: the actual difference

Continuous integration means every code change is automatically built and tested as soon as it's pushed. The goal is to find integration problems quickly, when they're cheap to fix, rather than at the end of a sprint when you're trying to merge two weeks of diverged work.

Continuous delivery means every change that passes CI is automatically prepared for release. A human still decides when to deploy. Continuous deployment goes one step further: if the tests pass, it ships automatically, no human intervention required.

Most teams do CI. Fewer do CD properly. Very few do continuous deployment, and that's fine — it requires a high degree of confidence in your test coverage and deployment safety mechanisms.

What a typical pipeline does

  1. Trigger — a push or PR fires the pipeline
  2. Install dependencies — restore from cache if available
  3. Lint and type-check — fast static analysis, fail early
  4. Unit tests — isolated, fast, no external dependencies
  5. Integration tests — hit real databases, services, slower
  6. Build — compile, bundle, produce an artifact
  7. Security scan — dependency vulnerabilities, secret detection
  8. Deploy to staging — on merge to main
  9. Smoke tests — verify the deployment actually works
  10. Deploy to production — manual trigger or automatic

Not every pipeline needs every step. A small internal tool doesn't need the same pipeline as a payment system. Over-engineering a pipeline is a real failure mode.

Feature flags

Feature flags let you deploy code to production without making it visible to users. You merge the code but wrap the new behaviour in a flag check — when you're ready to release, you flip the flag.

This is what makes trunk-based development practical. You don't need long-lived feature branches because the work-in-progress code goes to production behind a flag. You can test in production with a small percentage of users before full rollout.

The technical debt warning Feature flags accumulate. Every flag is a conditional branch that needs to be cleaned up after launch. Teams that never clean up old flags end up with code full of dead branches and nobody knows which flags are still active. Treat flag removal as part of shipping the feature.

Deployment strategies

Rolling deploy

Replace instances one at a time. At any point during deployment, some instances are running the old version and some the new. Simple, works well for stateless services. Problematic if the new version is incompatible with requests the old version generated.

Blue/green

Run two identical production environments. Deploy to the inactive one (green), run smoke tests, then switch traffic. Instant rollback by switching back. Costs double the infrastructure. Worth it for high-stakes deployments.

Canary

Route a small percentage of traffic to the new version, monitor for errors, gradually increase. The best balance of safety and resource cost for most teams. The tricky part is defining what "bad" looks like in your monitoring so you know when to stop the rollout.

When pipelines go wrong

A flaky test is a test that sometimes passes and sometimes fails without any code change. Flaky tests are worse than no tests because they teach engineers to ignore failures. When a test fails, people should never think "oh it's probably just flaky". Fix flaky tests immediately or delete them.

A pipeline that takes 45 minutes to run is a pipeline that people work around. They batch changes to reduce pipeline runs, they skip the pipeline locally, they merge without waiting. Invest in parallelisation, caching, and pruning slow tests. Ten minutes is a reasonable target for feedback on a typical pull request.

Healthy pipeline habits

  • Pipeline runs in under 10 minutes
  • Failed pipeline blocks merge to main
  • Secrets managed by CI environment variables, not hardcoded
  • Artifacts versioned and reproducible
  • Rollback takes under 5 minutes

Pipeline antipatterns

  • Skipping CI on "urgent" hotfixes
  • Flaky tests left unfixed for weeks
  • Deploy scripts that only one person understands
  • No staging environment before production
  • Manual steps in the middle of an automated pipeline
