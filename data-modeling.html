Data Modeling

Star schemas, normalization, slowly changing dimensions. The decisions you make here shape every analysis your team will ever run — ignore them and your warehouse will drift into chaos.

Why this matters more than most people think

Bad data models don't stop working immediately. They accumulate problems slowly — duplicated logic across dozens of queries, metrics that are almost-but-not-quite consistent across dashboards, analysts who won't trust a number until they've manually checked it. By the time you notice, unpicking the mess is a multi-month project.

Good data modeling is mostly about making the right tradeoffs early, documenting why, and being consistent about a small number of conventions.

Normalization vs denormalization

Normalized data (OLTP style) minimizes redundancy. Each fact lives in one place. You join tables to get the full picture. This is right for transactional systems where data changes frequently — you update a user's name in one row, not five hundred.

Denormalized data (OLAP/warehouse style) copies data into wide tables optimized for analytical queries. Fewer joins, faster aggregations, easier to understand. This is right for data warehouses. The redundancy is acceptable because analytical data is usually append-only or updates infrequently.

The star schema

The most common warehouse pattern. A central fact table contains measurable events (orders, page views, payments) with foreign keys to dimension tables that contain descriptive attributes (users, products, dates).

-- Fact table: one row per order
fact_orders:
  order_id, user_id, product_id, date_id,
  quantity, revenue, discount_amount

-- Dimension tables
dim_users:    user_id, name, country, signup_date, plan
dim_products: product_id, name, category, unit_price
dim_dates:    date_id, date, week, month, quarter, year, is_weekend

Queries on a star schema are predictable: join the fact table to whichever dimensions you need, aggregate. No nested subqueries hunting through a normalized schema.

Slowly changing dimensions

What happens when a user changes their plan, or a product changes category? The dimension value at event time was different from the current value. This is the "slowly changing dimension" problem.

  • SCD Type 1 — overwrite. Simple, but you lose history. "What was this user's plan when they made this purchase?" becomes unanswerable.
  • SCD Type 2 — add a new row for each change, with valid_from and valid_to dates. You can answer historical questions accurately, but the dimension table grows and joins get more complex.
  • SCD Type 3 — add a column for the previous value. Only useful if you care about exactly one version of history.

Most warehouses that care about attribution accuracy use Type 2 for key dimensions like users and products. It's more work to set up and maintain, but the alternative is analytics that silently give wrong answers.

The dbt approach

dbt (data build tool) has become the standard way to build warehouse models. You write SQL SELECT statements, dbt handles the CREATE TABLE / CREATE VIEW and dependency management. Models reference other models with {{ ref('model_name') }}, which builds a DAG.

-- models/marts/orders.sql
SELECT
  o.order_id,
  o.created_at,
  u.name AS user_name,
  u.country,
  o.revenue
FROM {{ ref('stg_orders') }} o
LEFT JOIN {{ ref('stg_users') }} u USING (user_id)

The layer pattern that works: raw (source data, untouched) → staging (cleaned, renamed, typed) → intermediate (joined, lightly transformed) → marts (business-facing, wide, documented). Jump straight from raw to mart and you'll regret it within six months.

Common data model mistakes Using production database IDs as warehouse keys without checking for reuse. Storing JSON blobs in a column instead of modeling the nested structure. Fan-out joins that silently multiply rows (always check row counts after a join). Metrics defined in models without documentation of the exact business logic — six months later nobody agrees on what "active user" means.
