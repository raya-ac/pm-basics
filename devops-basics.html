DevOps Basics

The culture, practices, and tools that unite development and operations.

devops ci-cd infrastructure automation

What is DevOps?

DevOps is a cultural and professional movement that stresses communication, collaboration, and integration between software developers and IT operations. It is not a single tool or role -- it is a philosophy that breaks down silos between the teams that build software and the teams that run it.

The core insight: when dev and ops work in isolation, software ships slowly and breaks often. When they share ownership of delivery and reliability, everything improves -- speed, stability, and trust. The goal is short feedback loops from idea to production.

The CALMS framework captures the five principles of healthy DevOps culture:

C

Shared ownership of outcomes. Blame-free post-mortems. Empathy across dev and ops.
A

Automate builds, tests, deploys, and infrastructure provisioning to eliminate toil.

Core DevOps Practices

Continuous Integration (CI)

Developers merge code changes frequently -- ideally multiple times per day. Each merge triggers an automated build and test suite. Bugs are caught early, before they compound into hard-to-debug regressions.

Continuous Delivery (CD)

Every change that passes CI is automatically prepared for release. The deploy to production may still require a manual gate, but the software is always in a releasable state. Continuous Deployment removes that gate entirely.

Infrastructure as Code (IaC)

Servers, networks, and databases are defined in code -- not via manual click-ops. Tools like Terraform and AWS CloudFormation let teams version-control infrastructure, review changes in PRs, and recreate environments reproducibly.

Monitoring and Alerting

Production systems emit metrics, logs, and traces. Alerts fire when thresholds are breached. This gives teams real-time visibility into health and performance so issues are caught before users notice them.

Incident Management

A defined process for detecting, responding to, and learning from production incidents. Includes on-call rotations, runbooks, severity levels, dedicated incident channels, and blameless post-mortems that produce systemic improvements.

DORA Metrics: Measuring DevOps Performance

The DORA research program (DevOps Research and Assessment) identified four metrics that reliably predict software delivery performance. Elite teams optimise all four simultaneously -- high frequency, short lead time, low failure rate, fast recovery.

DORA #1
Deployment Frequency
How often code reaches production. Elite: multiple deploys per day, on demand.
DORA #2
Lead Time for Changes
Commit to production time. Elite: under one hour. Reflects pipeline and review speed.
DORA #3
Change Failure Rate
Percentage of deploys that cause a production incident. Elite target: below 5%.
DORA #4
Time to Restore Service
Recovery time from a production failure. Elite: service restored in under one hour.

Use these as lagging indicators of process health, not targets to game. Genuine improvement requires systemic change: better test coverage, smaller PRs, faster pipelines, clearer runbooks.

For PMs: Working with DevOps Teams

What to understand:

  • DevOps investment is an enabler of speed, not overhead. A faster, reliable pipeline means your features ship sooner and break less often.
  • Infrastructure and tooling work rarely appears in the product backlog but is critical to long-term velocity. Advocate for it getting real sprint capacity.
  • DORA metrics are the PM-friendly window into engineering health. Ask your EM to share them in quarterly planning reviews.
  • On-call burden is a product concern. Excessive alerts from your features burn out engineers and slow delivery -- treat it as a quality issue worth prioritising.

How to collaborate effectively:

  • Loop in DevOps or SRE engineers early when a feature has significant infrastructure implications -- new data stores, third-party integrations, or high-traffic endpoints.
  • Define non-functional requirements up front: target uptime SLA, max p99 latency, expected peak request rate. Don't leave these as unstated assumptions.
  • Make post-mortems blameless by default. Your behaviour as PM sets the tone -- focus on systemic fixes, not individual mistakes.
  • Before every major launch, ask: "What is our rollback plan?" and "How will we know if this is failing within the first 30 minutes?"
