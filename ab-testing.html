A/B Testing

Statistical significance, sample sizes, novelty effects, and the many ways a well-run-looking experiment can still prove nothing.

What a test actually tells you

An A/B test measures whether a change causes a difference in a metric, with enough statistical confidence that the difference probably isn't random noise. That's all. It doesn't tell you why users behave differently, it doesn't generalise to users you haven't tested on, and it says nothing about long-term effects.

Most teams run tests to confirm decisions they've already made. This isn't wrong on its own — the risk is interpreting results to fit a conclusion rather than letting the test genuinely inform one.

The statistics you need to know

Statistical significance (p-value)

A p-value of 0.05 means: if there were actually no difference between variants, you'd see a result this extreme or more extreme 5% of the time by chance. It does not mean "there's a 95% chance the result is real." This misinterpretation is genuinely common, including among people who run experiments for a living.

Statistical power

Power is the probability of detecting an effect if one actually exists. Low power means you'll frequently miss real improvements. Most teams under-power their tests, run them too short, and then conclude "no effect" when the truth is "we couldn't detect an effect with this sample."

Minimum detectable effect (MDE)

Before you run a test, decide the smallest improvement worth detecting. A 0.1% conversion lift on a low-traffic page requires an enormous sample. A 5% lift on a high-traffic checkout page is detectable in days. Calculate sample size before starting, not after.

-- Rough sample size rule of thumb:
-- n ≈ 16 × σ² / δ²
-- where σ is variance, δ is minimum detectable effect
-- Most experiment platforms calculate this for you.
-- The point: run the calculation BEFORE launching.

Common ways tests go wrong

Peeking

Checking results daily and stopping as soon as p < 0.05 inflates your false positive rate dramatically. You need to pre-specify your sample size and run the test to completion. If this is genuinely impractical, use sequential testing methods that account for early stopping.

Multiple comparisons

Testing 10 metrics and reporting the one that was significant gives you a roughly 40% chance of a false positive even at p=0.05. Pre-specify your primary metric. Secondary metrics are exploratory, not confirmatory.

Novelty effect

Users engage with new things just because they're new. A redesigned feature might show a conversion lift for two weeks because users are paying attention, not because it's better. If your test window is too short, you're measuring novelty. Run tests long enough to see behaviour stabilise — at least one full week to capture weekly seasonality, longer for low-frequency actions.

Network effects and interference

If users interact with each other (social features, marketplaces, multiplayer), random assignment to control and variant contaminates the results. A user in control who interacts with a variant user has been indirectly treated. Cluster-based randomisation helps but is more complex.

The winner's curse The effect size you measure in an experiment is almost always an overestimate, especially for tests that barely reached significance. This is called the winner's curse. When you roll out the "winning" variant to 100% of users, the actual lift is usually smaller than the test suggested. Plan for this.

Good experiment hygiene

  • Pre-register hypothesis and primary metric
  • Calculate sample size before launch
  • Run for at least one full week
  • Check for sample ratio mismatch (SRM)
  • Segment results to check for heterogeneous effects

How teams fool themselves

  • Stopping as soon as p < 0.05
  • Testing 8 variants simultaneously
  • Reporting secondary metrics when primary is flat
  • Running the same test twice after a null result
  • Calling a 51/49 split "statistically tied" and shipping anyway
